## 1. 収益化チェックリスト(ここが終わるまで売上は0)

### 1-1. 独自ドメイン移行

github.io サブパスのままでは SEO・ASP 審査・ブランドのすべてで不利。最初にやる。

1. ドメインを取得(¥1,000〜2,000/年程度。`.com` / `.jp` / `.dev` など)。
2. Vercel にリポジトリを import し、カスタムドメインを割当(`vercel.json` でセキュリティヘッダ付与済み)。
3. 環境変数を2つ設定するだけで移行完了(README「デプロイ」参照):
   - `SITE_URL=https://取得したドメイン`
   - `SITE_BASE=`(空文字)
   → canonical / sitemap / 内部リンク / robots.txt / llms.txt / OG画像URL / CI の内部リンク検査
   まで**すべて自動追従**(手動更新は不要。robots.txt はビルド時生成、CI は canonical から base を導出)。
4. 旧 GitHub Pages からは可能なら移転案内(そのまま放置しない)。

### 1-2. ASP 申請(2〜3社に並行申請)

対象: **バリューコマース / A8.net / アクセストレード**(転職案件が厚い3社)。1社に絞らない —
同一オファーでも承認可否・単価・EPC が ASP により違う。

流れ(各社共通):
1. 会員登録 → サイト情報を登録(独自ドメイン後が通りやすい)。
2. 転職エージェント系オファー(レバテックキャリア / ビズリーチ / doda / Geekly 等)へ提携申請。
3. 審査は数日〜数週間。**否認されても理由を見て再申請すればよい**(記事追加後に通ることは普通にある)。

申請時のサイト紹介文(このサイトの強みをそのまま書く。誇張しない):

> エンジニア・IT転職に特化した比較・解説サイトです。現在 20本の記事(比較・選び方・
> 年代別・職種別)を公開しており、全ページに広告掲載の開示バナーとアフィリエイト
> ポリシー(/affiliate-policy)・開示ページ(/disclosures)を設置、外部リンクには
> rel="sponsored" を付与しています。捏造レビュー・架空体験談は掲載しない編集方針で、
> 公開前に景表法・ステマ規制の観点の機械チェックと目視確認を行っています。

- 開示体制(ステマ規制対応)と記事数を明示するのがポイント。審査側が見るのはそこ。
- 著者情報(`src/data/site.ts`)は実在の内容にしておく。無いものは書かない。

### 1-3. 承認後のリンク挿入(作業は1箇所)

記事側は全 CTA が `[テキスト](offer:<id>)` で書かれており、**リンクの実体は
`src/data/offers.mjs` に一元化**されている。承認が下りたら:

- **方法A**: `src/data/offers.mjs` の該当オファーの `url` にアフィリエイトリンクを記入。
- **方法B**: Vercel の環境変数 `OFFER_URL_<ID>`(例: `OFFER_URL_LEVTECH`)に設定(こちらが優先)。

どちらでも、次のビルドで**そのオファーの全記事の CTA が一斉に実リンク化**される。
記事を1本ずつ開いて置換する作業は存在しない。未承認のオファーは `href="#"` のまま公開され続ける(害なし)。

反映後の確認: `python3 scripts/compliance-lint.py --root src/content/posts` → build → 人間の目視。

## 2. 計測(数字を見ずに運用しない)

### 2-1. GA4 の affiliate_click イベント

`PUBLIC_GA_ID` 設定済み+訪問者の同意後に、CTA クリックが `affiliate_click` イベントとして飛ぶ。
パラメータ: `offer_id`(どのオファーか)/ `link_url` / `link_text` / `active`(`true`=実リンク、`false`=プレースホルダ)。

見方: GA4 →「レポート → エンゲージメント → イベント → affiliate_click」。
`offer_id` / `active` をカスタムディメンション登録すると内訳が出せる(探索レポートでも可)。

### 2-2. EPC の計算

**EPC = 発生報酬 ÷ アフィリエイトクリック数**。このニッチの相場は ¥160〜200/click
(STRATEGY.md §2)。自サイトの実測 EPC がこれを大きく下回るなら、記事とオファーの
intent がずれている。クリック数は GA4、発生報酬は各 ASP 管理画面から取る。

### 2-3. 承認前でも需要は測れる

`active: "false"` のクリック = プレースホルダ CTA を押した人。つまり **ASP 承認前から
「どの記事の・どのオファーに需要があるか」が分かる**。承認が下りたら `false` クリックの
多いオファーから優先的にリンクを入れる。同意率の分だけ実クリックより少なく出る点は割り引いて読む。

## 3. 週次運用ループ(1週間の型)

1. **記事2本**を追加(高intentを優先: 比較・評判・年代×職種)。lint → 目視 → 公開。
2. **内部リンク**: 新記事⇔既存記事・ピラーを相互に張り、frontmatter に `tags` を付ける
   (関連記事はタグで自動選定。orphan は CI の `check-orphans.mjs` が機械検出する)。
3. **Search Console**: 表示回数/掲載順位の伸びた query を確認。伸びかけの記事に追記。
4. **IndexNow 送信**: デプロイ後に `node scripts/indexnow-submit.mjs`(初回はスクリプト冒頭の
   キーファイル設置手順を実施)。Google には効かないので sitemap + Search Console が本線。
5. **既存記事の改稿**: 実体験・新情報を追記したら frontmatter の `updatedDate` を更新する
   (sitemap の lastmod と構造化データに自動反映。**実改稿なしでの日付更新はしない**)。
6. 外部リンク切れは週次の GitHub Actions(`link-check.yml`)が自動検出して Issue を立てる。
   性能を確認したいときは Chrome DevTools の Lighthouse を手動実行(恒常CIは不採用)。
7. §4 の KPI テーブルに1行追記。

これで週 3〜5時間。記事2本が最重量で、ここをサボると他の全部が無意味になる。

## 6. 月100万円への段階(規模感を直視する)

STRATEGY.md §2 の試算(EPC ¥160〜200/click 前提):
**月¥10万 ≈ 約20,800 PV/月 ≈ 約56記事。月¥100万 ≈ 約208,000 PV/月 ≈ 約556記事相当。**

| 段階 | run-rate | 必要になるもの(目安) |
|---|---|---|
| 第1段階 | 月¥10万 | 記事50〜60本・勝ち記事数本が上位表示・ASP直交渉で単価改善。個人の継続作業圏内 |
| 第2段階 | 月¥30万 | 記事150本級 + 更新運用。外注/ツール化なしでは執筆が律速。リスト(メール/LINE)で再訪導線 |
| 第3段階 | 月¥100万 | 記事500本級 + 編集チーム or 別レバレッジ(特単・自社案件・横展開サイト)。個人の片手間では構造的に届かない |

到達を保証するものではない。大半のサイトは第1段階の手前で更新が止まって終わる —
逆に言えば、§3 のループを止めないこと自体が最大の参入障壁になる。
本線はまず**累計¥100万(run-rate 月10〜15万)**であり、月100万は事業化の判断をしてから考える。
